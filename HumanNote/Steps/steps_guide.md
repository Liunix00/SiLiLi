# Steps 撰写指南

本指南帮助你正确编写 Steps 文件，使「司理理」Agent 能准确解析你的工作记录并自动更新项目的 plan 和 progress。

---

## 文件组织

```
HumanNote/Steps/{年}/{人名}/{日期}.md
```

- 每人每天一个文件（可跨天积累后一次性提交）
- 人名与文件夹名一致，Agent 通过路径自动识别作者
- 日期文件名建议使用 `YYYY.M.D.md` 格式，如 `2026.3.31.md`

---

## 日期标题

每天的记录以二级标题开始：

```markdown
## **2026.4.3**
```

支持在日期后添加备注：

```markdown
## **2026.4.3（周末）**
## **2026.3.28（加班）**
```

**要求：**
- 必须是 `##` 二级标题
- 日期格式：`YYYY.M.D`（月/日不补零）
- 允许的分隔符：`.` `-` `/`
- 加粗 `**` 可选但推荐

---

## 项目标签

在日期标题下，每条记录以项目标签开头：

```markdown
- 【001-问津】解析复旦科室和地区科室排行榜，优化数据库字段
- 【002-灵枢】写 AI 最佳实践指南文档
```

**要求：**
- 格式：`- 【NNN-项目名称】` 或 `- [NNN-项目名称]`（中英文括号均可）
- 编号必须是三位数字，如 `001`、`003`、`000`
- 项目名称必须与项目库中的全称一致
- 标签后紧跟该项目当天的工作描述

---

## 动作标记

在项目条目下方，用缩进的子行记录具体动作。缩进至少 4 个空格：

```markdown
- 【001-问津】刷公众号
    - done 完成数据清洗
    - todo 优化查询速度，3天
    - idea 某个技术方案可能有帮助
```

### 7 种标记

#### 1. `done` — 完成

标记某项任务已完成。Agent 会更新 plan 中该任务的状态。

```markdown
    - done 完成数据清洗
    - done                    ← 不写内容表示该项目条目整体完成
```

#### 2. `todo` — 待办

新增或记录待办事项。支持可选的工时和累加参数。

```markdown
    - todo 优化查询速度，3天
    - todo AI总结功能，输出总结实录，5天
    - todo 进行医院匹配，星期匹配，就诊人匹配 0.5天
    - todo debug，0.5 天
    - todo 确定进入TIPS_DATABASE的数据范围，2天
    - todo 进行                ← 最简写法，仅描述
```

**参数格式：** `todo 描述，工时，add`
- 工时格式：数字 + `天`/`d`/`日`，如 `3天`、`0.5天`
- `add` 表示累加工时（而非替换）
- 各参数用中文逗号 `，` 或英文逗号 `,` 分隔

#### 3. `blocked by` — 阻塞

标记项目被某个外部因素阻塞。

```markdown
    - blocked by 第三方接口未开放
    - blocked by 财务部审批流程
```

#### 4. `unblock` — 解除阻塞

标记之前的阻塞已解除。

```markdown
    - unblock 第三方接口已开放
```

#### 5. `plan` — 新计划

添加新的计划任务到 plan 文档中。支持结构化参数。

```markdown
    - plan 任务名称：用户权限系统，负责人：张三，工期：5d，ddl：4/15
    - plan 重构数据库索引
```

**结构化参数：**
- `任务名称：xxx` 或 `任务描述：xxx`
- `负责人：xxx`
- `工期：xxx`
- `ddl：xx/xx`
- 用中文逗号分隔

#### 6. `idea` — 灵感

记录想法和灵感，会被提取到项目的 idea 文件中。

```markdown
    - idea https://github.com/xxx 这个框架可能对检索架构很有帮助
    - idea plan和progress需要有历史版本，方便出周期性报告
    - idea 可以去看youtube
```

#### 7. `risk` — 风险

标记项目风险，Agent 会在 progress 中记录。

```markdown
    - risk 小红书信息爬取受限
    - risk 第三方 API 稳定性不足
```

---

## 各标记对文档的影响

| 标记 | Plan | Progress | Idea |
|------|------|----------|------|
| `done` | 更新对应任务状态为已完成 | 记录完成事项 | — |
| `todo` | 新增/更新待办任务及工时 | 记录新增待办 | — |
| `blocked by` | 标记任务为阻塞状态 | 记录阻塞信息 | — |
| `unblock` | 解除阻塞标记 | 记录解除阻塞 | — |
| `plan` | 新增计划任务 | 记录新计划 | — |
| `idea` | — | — | 按日期归档 |
| `risk` | — | 记录风险项 | — |

没有标记的子行会被视为普通备注（`note`），不触发特殊处理，但仍会作为上下文传递给 LLM。

## 完整示例

```markdown
## **2026.3.29（加班）**
- 【001-问津】根据收集小红书信息，构建分解，入库，查询，问答的全套流程。测试10个真正的用户问题
    - risk 小红书信息爬取受限
    - todo 确定进入TIPS_DATABASE的数据范围，2天
    - todo 完善文章分解入库的机制，3天
    - todo 优化查询速度，3天


## **2026.3.30**
- 【002-灵枢】向财务部交付水印pdf水印自动添加工具，需求按时完成
- 【002-灵枢】向人力资源部交付环评表自动和成工具，需求按时完成
- 【003-愈程】完成陪诊师诊前咨询助手prompt，实践使用Claude模型产生回复
    - todo 长任务需要技术组提供回调接口，联调长预计0.3天


## **2026.4.3（周末）**
- 【001-问津】刷公众号
    - idea https://github.com/xxx 这个新的组织图式可能对于检索架构很有帮助
- [000-知几] 和马万里讨论
    - idea plan和progress需要有历史版本，方便出周期性报告
    - idea 可以去看youtube
```

## 常见错误

| 错误 | 正确 |
|------|------|
| `### **2026.4.3**` （三级标题） | `## **2026.4.3**` |
| `- [001]问津` （缺少连字符） | `- 【001-问津】` |
| `- 001-问津 xxx` （缺少括号） | `- 【001-问津】xxx` |
| `  - done xxx` （缩进只有2格） | `    - done xxx` （至少4格） |
| `- done xxx` （没有缩进，在项目级） | `    - done xxx` （必须在项目条目下方缩进） |
| `- 【01-问津】` （编号不是3位） | `- 【001-问津】` |

---

## 简单规则速记

1. **日期用** `## **YYYY.M.D**`
2. **项目用** `- 【NNN-全称】描述`
3. **动作缩进 4 格**，以 `done` / `todo` / `blocked by` / `unblock` / `plan` / `idea` / `risk` 开头
4. 项目编号必须 **3 位数字**，名称必须和项目库 **完全一致**
5. `todo` 的工时用逗号分隔写在描述后面，如 `todo 任务描述，3天`
